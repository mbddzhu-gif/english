## 目标
对当前应用进行完整的 Vercel 部署可行性分析，清除阻塞项并落实一组可验证的优化（构建/路由/环境变量/静态资源/代码分割），最终完成可稳定运行的生产部署，并产出部署报告与后续维护建议。

## 当前结论（基于仓库现状的初步盘点）
- 前端不是 Next/React/Vue 等框架：纯 `index.html` + 原生 `js/*.js` + `css/style.css`。
- 已存在 `api/` 目录的 Vercel Functions 与 `vercel.json`，具备静态站点 + Functions 的部署基础。
- 发现明确部署阻塞项：
  - 前端会请求 `GET /api/xunfei/auth-ise` 与 `GET /api/xunfei/auth-iat` 获取讯飞 WS 鉴权 URL，但 `api/` 下当前没有对应函数实现，线上会 404，相关语音功能不可用。
  - Functions 依赖 `MINIMAX_API_KEY`（环境变量）才能工作；未配置会鉴权失败。
  - `server.js` 存在硬编码敏感信息（即使 Vercel 不运行它，仓库层面仍是高风险）。
  - 仓库包含 `node_modules/` 会导致上传体积与构建时间变差。

## 执行策略
分两条路径并行评估，默认执行“最小改动可上线”并在其上叠加“可控的构建与性能优化”：
- 路径 A（最小改动可上线）：保持静态站点形态（不引入框架/不做大规模重构），补齐 Functions、路由与环境变量，确保生产可用。
- 路径 B（增强构建与性能）：引入轻量构建（Vite）实现压缩、Hash、可缓存静态资源、代码分割/懒加载，输出到 `dist/`，并与 `api/` Functions 共存。

## 计划步骤
### 1) 全面可行性检查（只读分析 + 形成问题清单）
- 项目结构与 Vercel 识别：
  - 确认是否为“Static + Functions”或需要 Framework Preset。
  - 确认 `vercel.json` 路由是否会误伤静态资源与 Functions。
- 构建与输出：
  - 检查 `package.json` 脚本、Node 版本、构建产物目录。
  - 决定是否引入 `build` 与 `outputDirectory`。
- 环境变量与密钥：
  - 盘点所有后端/第三方（MiniMax、讯飞）所需变量。
  - 确认前端不直接持有密钥；所有密钥由 Functions 读取。
- 依赖兼容性：
  - 检查 Functions 运行时对 Node API 的使用（fetch、crypto、ws 等）。
  - 识别仅本地开发使用的依赖并隔离。
- 性能与可维护性：
  - 静态资源缓存策略、图片加载策略、代码加载时机（defer/懒加载）。
  - Functions 冷启动/超时风险、错误处理与日志策略（不记录敏感信息）。

产出物：
- “部署阻塞项清单（必须修）”与“优化清单（可选/建议）”，并给出优先级与验收点。

### 2) 清除部署阻塞项（必须完成）
- 补齐缺失的讯飞鉴权 Functions：
  - 新增 `api/xunfei/auth-iat` 与 `api/xunfei/auth-ise`（或等价路径）实现。
  - 统一从环境变量读取讯飞所需配置（如 APPID、API_KEY、API_SECRET 等；具体以现有前端调用与 `server.js` 的参数为准）。
  - 返回前端期望的 JSON 结构（与 `js/xunfei.js` 的解析保持一致）。
- 移除/降级仓库敏感信息风险：
  - 将 `server.js` 中硬编码密钥移除：改为读取环境变量，或将该文件替换为不含任何真实密钥的示例/本地模板。
- 依赖与仓库卫生：
  - 从仓库移除 `node_modules/`，补齐 `.gitignore`/`.vercelignore`，确保构建与上传可控。
- 环境变量落地：
  - 生成 `.env.example`（不含真实值）与 Vercel 环境变量配置说明（Production/Preview/Development）。

验收点：
- Preview/Production 上语音相关接口不再 404，能够返回有效鉴权信息（不暴露密钥）。
- 未配置关键 env 时，Functions 明确返回可诊断的错误（避免静默失败）。

### 3) 路由、重定向与缓存策略（必须完成）
- 评估是否需要 SPA fallback：
  - 若存在“直接访问子路径需要回到 index.html”的需求，则在 `vercel.json` 添加 rewrite 到 `/index.html`（排除 `/api/*` 与静态资源扩展名）。
- 配置静态资源缓存头：
  - 对 `/css/*`、`/js/*` 设置合理 Cache-Control（若引入 hash 文件名则使用 immutable；否则使用较短 max-age）。
- 需要时补充重定向规则：
  - 例如强制 HTTPS、规范化路径、旧路径兼容（以实际页面链接为准）。

验收点：
- 直接访问任意业务路径（若启用 fallback）可正确返回页面。
- 静态资源具备可预期的缓存策略，重复访问加载更快。

### 4) 构建与包体优化（执行路径 B；若风险过高则回退路径 A）
- 引入 Vite（vanilla）：
  - 增加 `build`/`dev` 脚本，产物输出 `dist/`。
  - 将脚本加载调整为模块化（`type="module"`）并通过动态 import 实现代码分割。
- 代码分割与懒加载策略（不改变业务逻辑）：
  - 将语音/评测、图片生成等非首屏必须模块改为按需加载。
  - 首屏仅加载渲染与基础交互逻辑。
- 静态资源优化：
  - 动态创建的图片元素设置 `loading="lazy"`、合理的尺寸/占位。
  - 需要时加入预连接（preconnect）到第三方域名（不包含敏感信息）。

验收点：
- `npm run build` 可稳定产出 `dist/`，产物体积下降、首屏脚本减少。
- 功能不回归，核心路径可用。

### 5) 本地与预发布验证（必须完成）
- 本地验证：
  - 在本地以 Vercel 模式验证静态资源与 Functions 路由（使用官方本地模拟方式）。
  - 覆盖关键用户路径：页面加载、生成内容、调用 MiniMax、语音相关功能（如启用）。
- 生产前检查：
  - 确认所有密钥仅存在于 Vercel 环境变量。
  - 确认日志不输出密钥与完整请求体（必要时脱敏）。

### 6) 部署到 Vercel（必须完成）
- 部署方式：
  - 方式 1：通过 Vercel Dashboard 连接 Git 仓库（推荐）。
  - 方式 2：使用 Vercel CLI（若环境允许）完成 `vercel` / `vercel --prod`。
- 部署后回归：
  - 验证静态资源、Functions、重定向/重写、缓存头是否生效。
  - 验证关键功能路径与错误边界。

### 7) 交付部署报告与维护建议（必须完成）
报告内容：
- 部署架构说明（Static + Functions / 是否引入构建）
- 环境变量清单（按环境划分）、密钥管理建议
- 路由/重定向/缓存策略说明
- 主要性能优化点与可量化指标（构建产物体积、首屏请求数/耗时的对比）
- 已知限制与后续优化建议（如进一步的模块拆分、API 速率限制、防滥用、监控告警）

## 风险与回退
- 若 Vite 模块化改造导致回归风险过大：
  - 回退到路径 A：不引入构建，仅做 defer、资源缓存头、图片懒加载与 Functions 修复，确保可上线。
- 若第三方（讯飞/MiniMax）鉴权流程需要更复杂的签名/时间戳校验：
  - 以 `server.js` 的现有实现作为签名基线（移除硬编码后迁移到 Functions），并补充单元/集成验证。

## 需要用户提供/确认的信息（仅在实施阶段阻塞时再确认）
- 讯飞相关：使用的服务类型（iat/ise）的正式参数、APPID/API_KEY/API_SECRET 的正确值与是否区分环境。
- 是否需要“子路径直达（SPA fallback）”能力。
- 生产域名与是否需要强制 HTTPS/自定义重定向。
